Connectors, Plugins, and MCP

By Bryce Murad · this.live

One of the most common misconceptions about AI agents is that they are brains floating in a vacuum. People imagine an LLM sitting in isolation, generating text, maybe answering questions. That is not an agent. That is a chatbot.

An agent is a brain with hands. Lots of hands. And those hands need to reach every system you care about.

The way this works in my setup is through three distinct layers of connectivity, and understanding the layers matters because it determines how extensible your agent system is and how fast you can add new capabilities.

The first layer is MCP, which stands for Model Context Protocol. Think of MCP like USB for AI. Any service that exposes an MCP server can be plugged into any agent that speaks the protocol. The agent discovers the available tools automatically, understands their schemas, and uses them as needed. It is a standardized interface that eliminates the need for custom integration code every time you want to connect a new service.

MCP is still early, but it is gaining momentum fast. The ecosystem includes connectors for databases, file systems, version control, search engines, and more. When I add an MCP server to my agent's configuration, it immediately knows how to use it. No retraining. No prompt engineering. Just new capabilities.

The second layer is native tools. These are built into the agent platform itself. My agents can run shell commands, read and write files, search codebases, manage processes, send messages, create cron jobs, generate images, analyze documents, and more. These do not require external services. They are part of the core platform.

This layer is where the raw power lives. Terminal access means an agent can build software, run tests, deploy services, and manage infrastructure. File access means it can read your codebase, edit configurations, and write documentation. Process management means it can start servers, monitor health, and restart failed services.

The third layer is custom API integrations. Telegram, Discord, Notion, GitHub, Linear, Google Workspace, Feishu. If a service has an API, the agent can use it. I have built custom connectors for the services I use daily, and the agent uses them seamlessly alongside native tools and MCP connectors.

The real magic is composition. An agent might search the web, extract content, write it to a file, commit it to git, and send me a Telegram summary. All in one continuous execution flow. The tools chain together naturally because the LLM is reasoning about which tool to use when, not following a predefined script.

This is why calling AI a chatbot misses the point entirely. It is a general-purpose computer operator. The LLM is the reasoning engine. The tools are the actuators. The more actuators you provide, and the better they are designed, the more the agent can do.

I think of it like hiring a brilliant generalist and giving them access to every department in your company. They can talk to sales, check inventory, write marketing copy, and fix a server. The only limit is what doors you give them keys to.

Build good tools. That is where the leverage is.
