Tool Calling: The Mechanism That Makes Agents Work

By Bryce Murad · this.live

Tool calling is the mechanism that makes agents actually useful, and most people do not understand how it works. Let me fix that, because once you understand tool calling, you understand why agents are fundamentally different from chatbots.

When you ask an agent to do something, the LLM does not execute anything directly. It cannot. It is a text generator running in a sandbox. It has no hands. So instead, it reasons about what needs to happen, then it generates a structured request to use a tool. The request includes the tool name and the parameters. The system receives this request, validates it, executes the actual tool, and returns the result to the LLM.

This loop repeats until the task is done. The LLM thinks. It calls a tool. It sees the result. It thinks again. It calls another tool. This is the agentic loop, and it is how an LLM goes from being a text generator to being an operator.

Here is a concrete example. You ask the agent to "find the latest version of our API documentation and update the README." The agent reasons: I need to find the docs, I need to read the current README, I need to compare them, and I need to write the updates. Then it starts calling tools.

First tool call: search the web for the API documentation. Result: a URL. Second tool call: extract the content from that URL. Result: markdown text. Third tool call: read the current README file. Result: the existing content. Fourth tool call: write the updated README. Result: success confirmation.

All of this happens in one continuous session. The LLM is reasoning at each step, deciding what to do next based on the previous result. It is not following a script. It is improvising. If the web search returns nothing useful, it tries a different query. If the README is missing, it creates one. If the documentation requires login, it uses browser automation instead of extraction.

The tools themselves are just functions. They can be anything. Read a file. Run a shell command. Search the web. Send a message. Query a database. Generate an image. The LLM does not need to know how the tool works internally. It just needs to know what the tool does, what parameters it takes, and when to use it.

This is why adding new capabilities to an agent is straightforward. You write a new tool function, you describe it in the agent's configuration, and the LLM automatically learns when to use it. No retraining. No fine-tuning. Just a new tool in the toolbox.

The quality of an agent is largely determined by the quality of its tools and the quality of its reasoning about which tool to use when. A great LLM with bad tools is crippled. A decent LLM with great tools is powerful. The magic is in the combination.

Build good tools. That is where the leverage is.
