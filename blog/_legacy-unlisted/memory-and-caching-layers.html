Memory and Caching: The Stack Nobody Talks About

By Bryce Murad · this.live

If you ask most AI builders about memory, they will talk about context windows. If you are lucky, they might mention RAG or vector search. But memory in a production agent system is not one thing. It is a stack of layers, and understanding the stack is the difference between a toy and a system that actually works at scale.

At the top of the stack is the context window. This is what the model sees right now, in this exact moment. It is the fastest layer, accessed at inference speed, but also the smallest. Once the context window fills up, old information gets pushed out, summarized, or truncated. This is where the active conversation lives. It is brilliant for immediate reasoning. It is terrible for persistence.

Below the context window is short-term session memory. These are things the agent writes down during a session that it might need to reference later in the same conversation. Maybe it looked up a file path, or calculated a value, or made a decision about how to structure a response. This survives within a single session but disappears when the session ends. It is scratch paper, not a filing cabinet.

Then there is persistent memory. Daily logs, long-term curated notes, user preferences, project context, relationship history. This survives across sessions and gets searched before every turn. This is where the agent remembers who you are, what you care about, and what you are trying to build. This is the layer that creates continuity. Without it, every conversation is a first date.

At the bottom of the stack is the cache. Not memory in the human sense, but a technical optimization layer. If the agent has already looked something up, computed something, or retrieved a document recently, the result might be cached. This avoids redundant work and redundant cost. Caching is invisible when it works and painfully obvious when it is missing.

Each layer has a different purpose, different speed, and different durability. The context window is for immediate reasoning. Session memory is for ongoing tasks within a conversation. Persistent memory is for long-term continuity across sessions. Cache is for efficiency and cost reduction.

A well-designed agent system uses all four layers intentionally. It does not try to shove everything into the context window. It does not rely solely on vector search. It does not ignore caching because "AI is non-deterministic anyway." It architects each layer for its specific job.

I see too many agent projects fail because they treat memory as an afterthought. They build brilliant reasoning, beautiful tool use, and sophisticated orchestration, and then they wonder why the agent feels forgetful, repetitive, and dumb.

It is not dumb. It just has no memory.

If you are building with agents and you only have one memory layer, you are making a tradeoff you probably do not want to make. Build the stack. Your users will notice.
