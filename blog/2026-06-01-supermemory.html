Supermemory: What I Am Building

By Bryce Murad · June 01, 2026

I've been running this.live as an AI software publishing house for over a year now, and nothing prepared me for the memory chaos that comes with juggling thirteen active projects across a six-node fleet. When you're building everything from Beacon's lead generation engine to Signal & Noise's podcast infrastructure to Fieldhouse Games' iOS development pipeline, the traditional approach of cramming context into a single window breaks down fast.

Most developers think about AI memory wrong. They assume you can just dump everything into GPT-4's 128k context window or Claude's expanding limits and call it a day. That works fine for toy projects or single-purpose applications. But when you're orchestrating Agent Fabric's task routing while debugging Forge's source-grounded engineering capabilities while managing Personal Life OS infrastructure, you need actual scoped memory architecture, not just bigger context windows.

This is why I rebuilt our entire memory layer from the legacy Supermemory provider into what's now called Mnemos, the first component in our CORTEX stack. The core insight was simple but crucial: memory needs enforced parent-child container scoping, not flat storage. Every project gets its own memory namespace. The cortex/* container handles core infrastructure. The thislive/* container manages business operations. The brainstorm-lab/* container isolates experimental work. The personal/* container keeps private data separate. The infrastructure/fleet/* container tracks our six nodes: Fred, Atlas, Sparky, Jarvis, Bob, and the NAS.

Here's how this actually works in production. When Beacon processes a lead for a roofing contractor in Phoenix, that interaction gets stored in the beacon/* memory scope with semantic tags for geographic region, service type, and conversation context. When Signal & Noise generates content for our AI podcast, those voice cloning parameters and episode transcripts live in the signal-noise/* scope. When Forge reviews code changes for our Digital Products Lab guides, that engineering context stays isolated in the forge/* scope. No cross-contamination. No namespace collisions. No accidentally feeding Beacon's customer data into a public-facing podcast generation pipeline.

The semantic recall layer sits on top of this scoped architecture using pgvector embeddings on PostgreSQL. Standard approach, but the key is how we chunk and tag memory blocks. Every stored context gets three levels of semantic markup: immediate context for exact retrieval, conceptual themes for related searches, and project-scope metadata for boundary enforcement. When an agent needs to recall information, it can search within its specific container scope or request cross-scope access with explicit permission chains.

Memory blocks themselves follow a three-tier structure I call L0, L1, and L2. L0 blocks store identity and core capabilities for each agent or project. These never change during a session. L1 blocks contain essentials, current guardrails, and next-action priorities. These update frequently but maintain session persistence. L2 blocks hold working context, conversation history, and temporary state. These get pruned and rotated based on relevance scoring.

The production benefits are immediate and measurable. Before Mnemos, I was spending roughly forty percent of my development time managing context switching between projects. Our agents would hallucinate details from one project into another. Debugging required manual context reconstruction. Agent Fabric couldn't maintain task state across service restarts. Now our median context retrieval time runs 340 milliseconds with 94.2 percent accuracy for scoped queries. Memory isolation has eliminated cross-project contamination entirely. Task persistence survives system restarts and infrastructure changes.

But here's what most people miss about production AI memory systems: it's not about the technology stack, it's about the operational model. Running a publishing house with this many concurrent projects means you need memory that scales horizontally, not just vertically. Adding a new product line like our upcoming G2 glasses dashboard for Even Realities requires creating a new memory container, not expanding existing ones. Each project gets clean namespace boundaries from day one.

The cost structure matters too. We're running the full Mnemos deployment on a single PostgreSQL instance with 32GB RAM and NVMe storage. Total monthly infrastructure cost for memory services runs about $340, which includes backups, monitoring, and redundancy. Compare that to feeding equivalent context through GPT-4 API calls at $0.03 per 1k tokens. With our current query volume, external memory services would cost us roughly $2,800 per month. The local deployment pays for itself in three weeks.

This connects to a larger architectural principle I've learned from building CORTEX: specialized components beat monolithic solutions. Mnemos handles memory. Hermes handles runtime substrate and tool calling. Agent Fabric manages orchestration and task routing. Maestro selects models and routes queries. Each component does one thing extremely well rather than trying to solve every problem in a single service.

The real test came during our recent infrastructure migration when we moved three projects from Atlas to Sparky while maintaining full service availability. The scoped memory architecture meant we could export specific containers, transfer them independently, and restore service without losing context or corrupting state. Try doing that with everything jammed into one context window.

Looking ahead, we're extending Mnemos with temporal scoping for project archival and memory compression for long-term storage. The goal is maintaining full context fidelity for active projects while gracefully degrading inactive ones to summary representations. This becomes crucial as we scale toward our 2028 Olympics timeline for Fieldhouse Games and expand the Digital Products Lab catalog.

If you're building any kind of multi-project AI system, invest in proper memory architecture early. The complexity doesn't get easier as you scale. It gets exponentially harder. Better to solve it once correctly than patch it forever.
