Security starts before the demo

By Bryce Murad · February 23, 2026

The boring rules that keep agents useful

Agents make security less optional

When agents can read files, call APIs, and make commits, security is not a checkbox at the end. It is the shape of the system. A careless permission boundary can turn a helpful assistant into a very fast mistake.

That is why I care about scoped memory, scoped tools, secret handling, and approval gates. Not because I enjoy bureaucracy. Because I have seen what happens when automation gets just enough access to be dangerous.

Secrets should not be content

API keys, tokens, credentials, and private customer data should not become model context by accident. The agent can know that a service exists without seeing the secret that authenticates it. It can propose a deployment without holding every production credential in its prompt.

Good systems make the safe path the default. Bad systems rely on everyone remembering the rules forever.

Trust, but gate the blast radius

I trust agents for a lot of work. I do not trust any worker, human or machine, with unlimited blast radius. Read access is different from write access. A draft is different from a deploy. A test command is different from deleting a database.

Security is not anti-speed. It is what lets you move fast without turning every success into a hostage situation.
