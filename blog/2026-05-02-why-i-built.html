Why I Built: What I Am Building

By Bryce Murad · May 02, 2026

I spent three months building a centralized real-time streaming platform before realizing I was solving the wrong problem. The wake-up call came when I calculated our infrastructure costs would hit $40,000 monthly at modest scale, all because I was routing every pixel through our servers. That's when I killed the entire architecture and started over with a local-first approach.

The original this.live was a WebRTC streaming service that promised ultra-low latency for creators and their audiences. Think Twitch but with sub-100ms delays instead of 3-6 seconds. I built it on Google Cloud Platform using Kubernetes clusters, deployed Janus WebRTC gateways across multiple regions, and created a React frontend that could handle thousands of concurrent viewers per stream. The technology worked beautifully in demos. The economics were catastrophic in reality.

WebRTC is designed for peer-to-peer connections, but streaming platforms need selective forwarding units to handle one-to-many broadcasting. Every viewer connection requires server-side processing power and bandwidth. At $0.08 per GB egress on GCP, a single 1080p stream with 1000 viewers would cost us $200 per hour just in bandwidth. Add the compute costs for transcoding, the database overhead for chat and analytics, plus the operational complexity of managing WebRTC state across distributed nodes, and we were looking at negative unit economics from day one.

I kept thinking I could optimize my way out of the hole. I spent weeks tuning the Selective Forwarding Unit configurations, implementing adaptive bitrate algorithms, and even exploring custom RTMP ingest pipelines to reduce processing overhead. I researched Cloudflare Stream, Amazon IVS, and Agora's APIs as potential cost-saving alternatives. Every path led to the same conclusion: centralized real-time streaming is fundamentally expensive because physics doesn't care about your business model.

The breakthrough came when I stopped thinking about this.live as a streaming platform and started thinking about it as developer infrastructure. Instead of building another Twitch competitor, what if I built the tools that let developers add real-time features to their own applications without the infrastructure nightmare I'd just lived through?

This realization coincided with my deeper dive into local-first software principles. I'd been following the work coming out of Ink & Switch, particularly their research on conflict-free replicated data types and operational transformation. The core insight is that modern applications can synchronize state between devices without depending on centralized servers for every interaction. SQLite running in WebAssembly, combined with sync protocols like those used in Linear or Figma, can deliver real-time collaboration while keeping user data local by default.

I redesigned this.live around four core primitives: local SQLite databases for each application instance, operational transform algorithms for conflict resolution, WebRTC data channels for direct peer-to-peer sync, and optional relay servers only when NAT traversal fails. Instead of streaming video through our infrastructure, developers can build applications where users collaborate directly with each other, using our SDK to handle the complex networking and synchronization logic.

The technical architecture centers on a TypeScript SDK that embeds wa-sqlite and provides hooks for React, Vue, and vanilla JavaScript applications. When a user makes a change, it gets written to their local SQLite database immediately, then broadcast to connected peers through WebRTC data channels. Other clients apply the changes using operational transform, ensuring consistency even when users edit the same data simultaneously while offline.

For developers who want to ship quickly, we provide hosted relay servers that facilitate peer discovery and handle WebRTC signaling. But the critical difference is that these servers never see application data. They're pure networking infrastructure, similar to STUN and TURN servers. This means our costs scale with connection establishment, not with ongoing data transfer or storage. A successful application built on this.live costs us pennies per month to support, not hundreds of dollars.

The business model flips from impossible to obvious. We open source the core SDK under MIT license, letting developers integrate real-time features into their applications for free. Premium features include managed relay infrastructure with global edge locations, monitoring dashboards for connection quality and sync performance, and enterprise-grade security features like end-to-end encryption key management. Developers bring their own backend services for authentication, billing, and any server-side logic their applications require.

I'm launching this.live as an open source project first, with a four-phase rollout over the next six months. Phase one focuses on developer experience: comprehensive documentation, example applications, and a smooth onboarding flow for JavaScript developers. Phase two adds framework-specific integrations, starting with Next.js and expanding to SvelteKit, Nuxt, and other popular stacks. Phase three introduces the premium managed services, and phase four expands beyond JavaScript to native mobile and desktop platforms.

The competitive landscape validates this approach. Yjs and Automerge provide excellent conflict-free data structures but require significant networking expertise to implement real-time sync. Socket.io and Pusher offer simple real-time APIs but force developers into centralized architectures with ongoing per-message costs. PartyKit and Liveblocks provide managed real-time infrastructure but keep all data on their servers. This.live combines the performance benefits of local-first architecture with the developer experience of managed infrastructure.

Building this.live taught me that sometimes the best solution isn't optimizing your original approach but questioning whether you're solving the right problem. Centralized real-time streaming works for companies with massive scale and venture capital. Local-first real-time collaboration works for everyone else. The future belongs to applications that respect user data ownership while delivering the seamless collaborative experiences users expect.
