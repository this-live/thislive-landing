Forge is about grounding, not magic coding

By Bryce Murad · March 16, 2026

Why coding agents need the source tree

The problem is not that models cannot code

Models can write code. That argument is over. The problem is that writing code is not the same thing as understanding a codebase. A patch can look reasonable, compile in isolation, and still break the contract the rest of the system depends on.

Forge exists because I got tired of that gap. I do not want an agent that guesses from vibes. I want one that reads the source tree, understands imports, respects tests, and knows when it does not have enough context.

Source grounded means receipts

The useful version of Forge is boring in the right ways. It should know which files it inspected. It should identify the symbols involved. It should explain the dependency it is about to touch. It should run the tests that matter. It should leave a trail a human can review.

That is slower than blindly generating a diff. Good. Speed without grounding just creates review debt.

The adapter path

Forge also points toward a longer-term loop: collect source-grounded examples, evaluate them, train adapters where it makes sense, and serve specialist behavior locally. I have scaffolding and readiness work here. I am not going to pretend a full cloud training flywheel has already run end to end. It has not.

The honest claim is still strong enough: better engineering agents start with better grounding.
