Agent Fabric: What I Am Building

By Bryce Murad · May 04, 2026

I've been watching the agent orchestration space evolve rapidly over the past six months, and one project keeps catching my attention: agent-fabric. While everyone else is building yet another chat interface or throwing more parameters at the same problems, the agent-fabric team is tackling something fundamentally different. They're building infrastructure for multi-agent coordination that actually works in production.

The core insight behind agent-fabric is that most AI applications will eventually need multiple specialized agents working together, not one monolithic model trying to do everything. This isn't a new idea, but the execution here is what matters. Instead of building another framework that looks good in demos but falls apart under real load, they've architected a system that handles the messy realities of distributed agent coordination.

At its heart, agent-fabric runs on a message-passing architecture built on top of Redis Streams. Each agent registers capabilities and subscribes to relevant event patterns. When work comes in, the fabric routes it to appropriate agents based on capability matching and current load. The routing layer uses a gossip protocol for service discovery, which means agents can join and leave the network dynamically without central coordination. This is crucial for production deployments where you need to scale individual agent types based on demand.

The implementation details get interesting when you dig into how they handle consensus and conflict resolution. Unlike my own maestro project, which uses quorum-based voting for critical decisions, agent-fabric takes a different approach. They implement a lightweight conflict resolution system based on capability scores and execution history. When multiple agents could handle the same task, the fabric considers both the agent's declared capability level and its recent success rate for similar tasks. It's simpler than a full consensus system but more sophisticated than basic round-robin routing.

What impressed me most is their handling of state management across agents. Each agent maintains its own local state, but the fabric provides a distributed state synchronization layer built on operational transforms. This means agents can work on shared data structures without explicit locking or complex coordination protocols. The synchronization layer handles merging concurrent updates automatically, similar to how collaborative editing works in modern document editors.

The economic model here is particularly clever. Running multiple specialized agents is theoretically more expensive than using one large model, but agent-fabric's routing intelligence means you can use smaller, cheaper models for most tasks and only invoke expensive models when necessary. In my testing with a mixed workload, I saw cost reductions of about 40% compared to using GPT-4 for everything, while maintaining similar output quality for most tasks.

The deployment story is where agent-fabric really shines compared to alternatives. The entire system runs as a set of Docker containers with sensible defaults. You can start with a single node running multiple agent types and scale out to a distributed cluster as your needs grow. The fabric handles service discovery and load balancing automatically. I had a three-node cluster running on AWS EC2 in under twenty minutes, including the time to provision instances.

Their approach to observability deserves special mention. Instead of black-box metrics, agent-fabric exposes the internal decision-making process of the routing layer. You can see why specific agents were chosen for tasks, how capabilities were scored, and where conflicts occurred. This transparency is essential for debugging production issues and optimizing agent configurations. The observability stack integrates with standard tools like Prometheus and Grafana, so it fits into existing monitoring infrastructure.

The security model addresses concerns that most agent frameworks ignore completely. Inter-agent communication uses mTLS by default, with certificate rotation handled automatically. Each agent runs in its own sandbox with carefully defined permissions. The fabric validates all capability declarations and can quarantine agents that behave unexpectedly. For production deployments handling sensitive data, these aren't nice-to-have features, they're requirements.

Performance characteristics are solid for a system this flexible. In my benchmarks, routing decisions typically complete in under 5ms, even with dozens of agents registered. The bottleneck is usually the agents themselves, not the coordination layer. Memory usage scales linearly with the number of active agents, with minimal overhead from the fabric itself. Network traffic between nodes stays reasonable thanks to efficient message batching and compression.

The project's approach to extensibility is pragmatic rather than over-engineered. New agent types register by implementing a simple interface and declaring their capabilities. The capability system uses structured tags rather than free-form descriptions, which makes routing decisions deterministic and debuggable. You can implement custom routing strategies by providing alternative scoring functions, but the defaults work well for most use cases.

One limitation I've noticed is that agent-fabric assumes relatively short-lived tasks. For agents that need to maintain long-running conversations or complex state, the current architecture isn't ideal. The team acknowledges this and has session management on their roadmap, but it's not available today. This makes it less suitable for applications like customer service bots that need to maintain context across extended interactions.

The open source nature of agent-fabric matters more than usual because agent orchestration is such a foundational layer. You need to understand exactly how routing decisions are made and be able to modify the behavior when your specific use case requires it. The codebase is well-structured and documented, making it feasible to contribute improvements or adapt it for specialized needs.

Looking forward, I expect agent-fabric to become increasingly relevant as more teams move beyond single-agent applications. The infrastructure layer they've built solves real problems that every multi-agent system eventually faces. While the space is crowded with frameworks and platforms, most focus on the AI models themselves rather than the coordination challenges. Agent-fabric fills a genuine gap in the ecosystem.
