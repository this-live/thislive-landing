Building With AI: What I Am Building

By Bryce Murad · June 08, 2026

I have spent the last year running a six-node agent fleet across three data centers and one private network. I built this.live as an AI software publishing house, and every month the architecture gets clearer about what matters and what does not.

The biggest mistake I see founders make is thinking the question is which model to use. It is never about the model. It is about the infrastructure that sits between the model and the real world. A model is a commodity. The systems that feed it context, route its outputs, and give it tools to act are what separate shipping products from shipping demos.

My fleet runs on a DGX Spark GB10 with 128GB of VRAM for local inference, Mac minis for coordination and engineering, a Synology NAS for shared storage, and a RTX 5080 workstation for GPU-intensive workloads. The total hardware cost is roughly $25,000. I spend about $2,000 per month on OpenRouter API calls for cloud models when local inference hits its limits. This infrastructure supports 13 active projects across Beacon, Maestro, Fieldhouse Games, Brokernomex, and other subsidiaries under the this.live umbrella.

The architecture that actually works has five layers, and each one earns its place.

The perception layer handles raw input transformation. Every agent in the fleet ingests text, voice, API responses, and sensor data. The trick is not the ingestion. It is the structuring. I use a combination of regex extraction for known schemas and LLM-based extraction for fuzzy patterns. The structured output feeds directly into the reasoning layer. Nothing leaves the perception layer unvalidated.

The reasoning layer is where planning, tool selection, and adaptive decision-making happen. I use ReAct patterns for simple task chains and plan-and-execute for multi-step workflows. The model generates a reasoning trace first, then decides which tools to invoke and in what order. The trace is not for the user. It is for debugging. When an agent fails, I read the trace and find the exact decision point that went wrong.

The memory layer is the one everyone underestimates. Most agent systems treat memory as an afterthought. I treat it as the foundation. Every agent has three memory types. Short-term conversational context lasts for the duration of a session. Long-term knowledge storage persists across sessions and covers project architecture, API conventions, and established patterns. Episodic memory captures specific events with temporal information. This is how agents learn from past failures. If an agent broke a deployment last Tuesday, it remembers that when it reaches for the same tool today. The memory system I built uses a local embedding backend with a bge-base-en-v1.5-compatible model, 384 dimensions, and semantic recall across 13 containers with zero cross-contamination between projects.

The tool execution layer connects agents to external systems, APIs, databases, and services. This is the layer where reliability matters most because tool failures cascade. Every tool invocation has retry logic with exponential backoff, input validation before execution, and structured error reporting back to the reasoning layer. I learned this the hard way when a failed Stripe API call broke the entire payment flow in Beacon. Now every external call has a circuit breaker pattern. If three consecutive calls fail, the tool enters a cooldown period and the reasoning layer switches to a degraded mode.

The orchestration layer coordinates flow between components and manages state across multi-step workflows. I use graph-based architectures inspired by LangGraph patterns but implemented in my own framework. The state graph supports persistence, resumable checkpoints, and human interruption points. This makes workflows debuggable. When a workflow hangs, I can inspect the graph state at any node, not just the current one. The state machine is serialized to SQLite for durability and replayed on recovery.

The knowledge retrieval layer uses hybrid search combining dense vector embeddings with sparse BM25 keyword matching. Results are merged through Reciprocal Rank Fusion and re-ranked with a cross-encoder before feeding back to the reasoning layer. The quality of this retrieval directly determines the accuracy of every agent decision. A bad retrieval returns context the model should ignore. A good retrieval returns the right document at the right time. I test this weekly by running a corpus of internal documents through the retrieval pipeline and measuring precision at various recall levels.

What keeps me up at night is not the technology. It is the discipline. Every time I add a new feature, I have to resist the urge to add a new agent. More agents means more surface area for failure, more state to manage, more debugging overhead. The best agent is the one that does three things extremely well instead of seven things adequately. This discipline extends to the infrastructure. The fleet runs on six nodes. Adding a seventh node for a single workflow is always the wrong decision.

The business side is where this infrastructure pays dividends. Beacon is the only revenue-producing product right now. It serves as the live revenue engine while Maestro, Fieldhouse Games, and the other subsidiaries continue development. The agent infrastructure that supports development also supports operations. Automated fleet audits, nightly PRD reviews, and daily blocker sweeps run on schedule without human intervention. This automation replaces what would have been three part-time operations roles.

The content pipeline that generated this post runs every morning at six AM. It pulls context from the Supermemory container for thislive, fetches the day's topic from a rotating calendar, checks git activity across the relevant repositories, and generates a blog post, a LinkedIn draft, and an X post. The blog posts run on a static site served from the same infrastructure that powers Beacon. The social posts queue through Buffer for scheduled delivery. This is not a content experiment. It is a demonstration of the infrastructure itself. Every word published is a system running.

I stopped fundraising in May after a meeting that went nowhere with Jason Calacanis. The pitch deck was incomplete and the narrative was not compelling enough to justify the time investment. This means every product under this.live has to earn its own way. The agent infrastructure is part of that strategy. By building internal tools that replace operations roles and accelerate product development, I reduce the burn rate while increasing the velocity of delivery. The infrastructure is not a cost center. It is a force multiplier.

The technology landscape shifts every month. New models launch, new frameworks appear, new infrastructure patterns emerge. The ones that matter are the ones that reduce failure surface, not the ones that add capability. An architecture that can survive a failed tool invocation, a model API outage, and a node going offline at the same time is more valuable than one that can do four additional tasks but breaks when any single dependency fails.

Building with AI is not about building with AI. It is about building systems that reliably deliver value to users while the model does its part. The model is one component in a larger machine. The machine is what ships products.
