The problem it kills

Agents that work alone — and step on each other

You have more than one agent, but they run as isolated chats. Nothing coordinates them, nothing isolates them, and nothing keeps a record. So they collide, repeat work, and leave you guessing what actually happened.

One agent at a time, in serial

Work that could fan out across five agents runs one chat at a time. The fleet you're paying for sits idle while a single thread grinds through a queue it could have parallelized.

No isolation, no record

Two agents touch the same files and clobber each other. And when a run goes sideways, there's no receipt — no record of who ran what, on which inputs, with what result. You can't audit what you can't see.

The dispatch path

One task in. A fleet of agents out — with a receipt for each.

A task enters the :3333 service and fans across agents, each running in its own git worktree. Progress streams over SSE; every node returns an ExecutionReceipt v1, and runs that route through Maestro carry a durable decision and cost.

  • Parallel by default. Lanes run at once in isolated worktrees — no clobbering, no serial queue.
  • Receipt for everything. Each node returns an ExecutionReceipt v1; the bus collects them.
  • Priced when routed. Maestro threads a decision_id + cost into the receipt.

Dispatch in practice

Fan one task across the fleet

Hand the service a task and lanes, and it dispatches each agent into its own worktree, streams progress over SSE, and hands back a receipt per node. Deterministic, auditable, parallel.

Boring on purpose. The spine doesn't try to be clever. It dispatches, isolates, and records — so the smart part, your agents, can run without stepping on each other.
