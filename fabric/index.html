The problem it kills

A fleet you own, running like one slow chat

You have many agents and several machines, but the work still flows through one lane on someone else's server. The nodes you own sit idle, your context lives in a silo you don't control, and when a run goes sideways there's no receipt to audit.

Serial by default

Work that could fan across the fleet runs one agent, one lane, one machine at a time. The nodes you own sit idle while a single thread grinds through a queue it could have parallelized.

Centralized and opaque

The orchestration lives on someone else's server, your context held in their silo. When a run goes sideways there's no receipt to audit — no record of who ran what, on which inputs, with what result. You can't audit what you can't see.

The dispatch path

One request in. A fleet of agents out — with a receipt for each.

A request enters the :3333 service on your hardware and fans across nodes, each lane running in its own git worktree. Progress streams over SSE; every lane returns an ExecutionReceipt v1, and the router threads a decision_id + cost into each one.

  • Parallel by default. Lanes run at once in isolated worktrees across the fleet — no clobbering, no serial queue.
  • Receipt for everything. Each lane returns an ExecutionReceipt v1; the durable bus collects them locally.
  • Explainable per lane. The router threads a decision_id + cost into the receipt.

Dispatch in practice

Fan one task across the fleet

Hand the service a request and lanes, and it fans each agent into its own worktree across the fleet, routes per lane, streams progress over SSE, and hands back a receipt per node. Decentralized, auditable, parallel.

Boring on purpose. The spine doesn't try to be clever. It dispatches, isolates, routes, and records — so the smart part, your agents, can run in parallel without stepping on each other.

fabric — dispatch
# dispatch a 3-lane parallel job to the execution service $ curl localhost:3333/dispatch -d '{ "graph": "parallel", "lanes": 3 }' → task t_8b41 accepted · 3 lanes · isolated worktrees # progress streams live over SSE $ curl -N localhost:3333/events/t_8b41 lane A running · lane B running · lane C running lane A done · lane B done · lane C done # every lane returns a durable ExecutionReceipt v1 { "task": "t_8b41", "state": "succeeded", "receipts": 3 }
