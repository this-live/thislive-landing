The memory model

Scoped by hierarchy, curated by promotion

Recall is project-scoped by default and labeled everywhere; the shared root is write-protected. Memory only climbs to a wider scope through an explicit promotion ritual.

01

Container hierarchy

Memories live in paths like cortex/<pillar>. Recall on a container returns its memories plus ancestors' promoted ones — never siblings.

02

Scoped & labeled recall

Unscoped recall floors to a triage scope; it is never a global scan. Every result carries its source container_tag.

03

Protected root + promotion

Direct writes to a shared root return 403. A memory reaches shared scope only via an explicit promotion ritual with an attestation.

04

Sessions + tombstones

Ephemeral session containers with TTL, atomic promote-on-close (exactly-once), and soft-delete tombstones that retain a version snapshot.

Real capabilities

What Mnemos actually does

Running in production today against a real NAS — every capability below is probed live and maps to a receipt in the ledger.

Project-scoped by default

Recall is bounded to a container and its promoted ancestors. Siblings never leak in, and an unscoped query floors to triage — never a global scan of the whole fleet.

Labeled cross-project recall

When recall does span projects, every row carries its container_tag so you always know where a memory came from. Provenance is never lost.

Write-protected root

A direct write to the shared root is rejected with a 403. Shared knowledge is earned through a promotion ritual — not dumped by any agent that feels like it.

Tombstones, not deletes

Soft-delete excludes a memory from every recall path while retaining a version snapshot — so a delete is auditable and never silently loses history.

Sessions, exactly-once

Ephemeral session containers create, store and recall scoped, then atomically promote-on-close — exactly once — and TTL-reap. No duplicate promotions on re-entry.
